# e-Gov 法令データ ドメインリファレンス

> 本文書は `lawapi-v2.yaml`（OpenAPI 仕様）を補完する目的で、YAML だけでは得られないドメイン知識を整理したものである。
> 情報源は e-Gov 法令データドキュメンテーション（α版）のスクレイピング済みドキュメント（`../docs-alpha/`）。
> API エンドポイントの仕様自体は `lawapi-v2.yaml` を参照のこと。

---

## 1. 法令種別

YAML では `law_type` enum として `Constitution | Act | CabinetOrder | ImperialOrder | MinisterialOrdinance | Rule | Misc` が定義されている。各値の意味は以下の通り。

| `law_type` | 和名 | 説明 |
|---|---|---|
| `Constitution` | 憲法 | 日本における最高位の法令 |
| `Act` | 法律 | 国会の可決により成立。太政官布告のうちデータ整備時点で法律として有効なものを含む |
| `CabinetOrder` | 政令 | 内閣が制定。太政官布告のうちデータ整備時点で政令として有効なものを含む |
| `ImperialOrder` | 勅令 | 旧形式の法令。データ整備時点で有効なもののみ |
| `MinisterialOrdinance` | 府省令 | 各機関の長等が定める法令 |
| `Rule` | 規則 | 各機関の長等が定める法令 |
| `Misc` | その他 | 上記に分類されない法令 |

**データ整備基準日**:
- 憲法・法律・政令・勅令: 2015(平成27)年8月1日時点
- 府省令・規則: 2016(平成28)年10月1日時点

**法令番号と効力の不一致**: 法令番号が示す種別と実際の効力が異なる場合がある。例:「出入国管理及び難民認定法」は「昭和二十六年政令第三百十九号」という法令番号を持つが、法律としての効力を有する（ポツダム命令に由来）。e-Gov では実際の効力に基づいて `LawType` が設定される。

> 詳細: `../docs-alpha/lawtypes-and-lawid.md`

## 2. 法令ID の内部構造

YAML では `law_id` を opaque な string として扱うが、内部は 15 桁の構造化された識別子である。

```
YYYTTTTTTTTTNNN
│  │        │
YYY 年（元号1桁 + 年数2桁ゼロ埋め）
   TTTTTTTTT 法令種別コード（9桁）
            NNN 法令番号（3桁ゼロ埋め）
```

**例**: 行政手続法（平成五年法律第八十八号）
```
405AC0000000088
│  │        │
405 → 平成(4) 05年
   AC0000000 → 法律（閣法）
            088 → 第八十八号
```

### 元号コード（1桁目）

| コード | 元号 |
|--------|------|
| `1` | 明治 |
| `2` | 大正 |
| `3` | 昭和 |
| `4` | 平成 |
| `5` | 令和 |

### 法令種別コード（4〜12桁目）

| 種別 | コード | 6〜12桁目の意味 |
|------|--------|------------------|
| 法律 | `AC` | `0000000`=閣法, `1000000`=衆議院議員立法, `0100000`=参議院議員立法 |
| 政令 | `CO` | `0000000`=政令, `1000000`=法律としての効力 |
| 勅令 | `IO` | `0000000`=政令, `1000000`=法律としての効力 |
| 太政官布告 | `DF` | `0000000`=政令, `1000000`=法律としての効力 |
| 太政官達 | `DT` | `0000000`=政令, `1000000`=法律としての効力 |
| 太政官布達 | `DH` | `0000000`=政令, `1000000`=法律としての効力 |
| 府省令 | `M1`〜`M6` | 関係府省等の組み合わせを表すビットフラグ（16進数7桁） |
| 人事院規則 | `R` + `JNJ` | 8〜9桁: 規則分類, 10〜12桁: 分類内連番, 13〜15桁: 改正連番 |
| その他規則 | `R` + 機関コード8桁 | 機関コード（下表）+ 番号3桁 |
| 内閣総理大臣決定 | `R` + `PMD` | 8〜11桁: 決定月日, 12〜15桁: 同日内連番 |

### 規則の機関コード（主要なもの）

| コード | 機関 |
|--------|------|
| `00000001` | 会計検査院 |
| `00000013` | 最高裁判所 |
| `00000014` | 衆議院 |
| `00000015` | 参議院 |
| `00000019` | カジノ管理委員会 |

> 完全な一覧: `../docs-alpha/lawtypes-and-lawid.md`

## 3. 法令番号

法令番号は法令の種類・制定者・年別に付される人間可読な識別子。YAML では `law_num` フィールドに格納される。

**形式**: `{元号}{年}年{法令種別}第{番号}号`
- 例: 「平成五年法律第八十八号」= 平成5年に公布された法律の88号

**府省令の特殊性**:
- 制定者名が含まれる（例: 「昭和六十年郵政省令第二十五号」）
- 共同命令は複数省庁名を含む（例: 「令和二年内閣府・総務省・経済産業省令第一号」）
- 制定当時の名称が残る（郵政省 → 現・総務省）

**例外**: 人事院規則は「人事院規則八―一二（職員の任免）」のように独自体系。

> 詳細: `../docs-alpha/lawtypes-and-lawid.md`

## 4. 法令履歴ID

YAML では `law_revision_id` として使用。構造は以下の通り。

```
[法令ID]_YYYYMMDD_[改正法令の法令ID]
```

| 要素 | 意味 |
|------|------|
| `[法令ID]` | 被改正法令の法令ID（15桁） |
| `YYYYMMDD` | 改正規定の施行日 |
| `[改正法令の法令ID]` | 改正法令の法令ID。初版・基準日時点は `000000000000000` |

**例**: `405AC0000000088_20231201_505AC0000000056`
- 被改正法令: 行政手続法（`405AC0000000088`）
- 施行日: 2023年12月1日
- 改正法令: 令和五年法律第五十六号（`505AC0000000056`）

> 詳細: `../docs-alpha/law-revisions.md`

## 5. 法令 XML 構造

YAML の `elm` パラメータおよび XSD（`../XMLSchemaForJapaneseLaw_v3.xsd`）は構文定義を提供するが、各要素の意味や運用上の注意は以下の通り。

### 5.1 基本階層

```
Law
├── LawNum          （法令番号: string）
└── LawBody
    ├── LawTitle    （法令名。Kana, Abbrev, AbbrevKana 属性あり）
    ├── TOC         （目次）
    ├── MainProvision（本則）
    │   └── Part/Chapter/Section/Subsection/Division
    │       └── Article（条）
    │           └── Paragraph（項）
    │               └── Item（号）
    │                   └── Subitem1〜Subitem10（号細分、最大10階層）
    ├── SupplProvision（附則）
    └── AppdxTable / AppdxNote / AppdxStyle / AppdxFig / AppdxFormat / Appdx
```

### 5.2 セマンティクス上の注意点

**本則 vs 附則**:
- 本則（`MainProvision`）: 法令の実体規定
- 原始附則: `<SupplProvision>` で `AmendLawNum` 属性なし。制定時の附則
- 改正附則: `<SupplProvision AmendLawNum="..." Extract="true|false">`。改正法令に属する附則であり、表示中の法令自体の附則ではない

**枝番号（Branch Numbers）**:
- 「第二十一条の二」→ `Num="21_2"`
- 「第三十八条の三の二」→ `Num="38_3_2"`
- 枝番号は独立した条文。「第二十一条」と「第二十一条の二」は親子関係にない

**Sentence の Function 属性**:
- `Function="main"`: 本文
- `Function="proviso"`: ただし書

**LawBody.Subject 属性**: 件名。題名のない法令のために設けられたが、現在はほぼ使用されない（`LawTitle` に登録）

**RelatedArticleNum**: 別表等がどの条文に関係するかを示す要素（例:「（第九条関係）」）

### 5.3 XSD と本文書の役割分担

| 情報の種類 | XSD | 本文書 |
|------------|-----|--------|
| 有効な要素・属性の列挙 | 完全 | — |
| 親子関係の制約（minOccurs, maxOccurs） | 完全 | — |
| 各要素の意味（日本語解説） | なし | あり |
| 運用上の使用実態（Subject は未使用等） | なし | あり |
| 枝番号のエンコーディング規則 | なし | あり |

> 要素ごとの詳細解説: `../docs-alpha/xml-schema.md`
> XML 構造と条文の対応: `../docs-alpha/law-structure-and-xml.md`

## 6. 改正・施行モデル

API のリビジョン管理機能（`/law_revisions`, `asof` パラメータ等）を正しく使うために必要なドメイン概念。

### 6.1 基本概念

| 用語 | 意味 |
|------|------|
| 公布（promulgation） | 官報掲載。法令の内容が確定する時点 |
| 施行（enforcement） | 法令の効力が発生する時点。公布と同日の場合もあれば、将来日の場合もある |
| 改正法令 | 他の法令を改正する法令（能動側） |
| 被改正法令 | 改正される法令（受動側） |
| リビジョン | 改正適用後の被改正法令の版 |
| 溶け込み | 改正規定を被改正法令のテキストに適用する操作。文字単位で改正箇所を特定する |

### 6.2 高度な施行パターン

**複数施行日**: 単一の改正法令が複数の施行日を持つ場合がある。各改正規定に異なる施行日が設定される。

**施行期日の不確定性**: 「政令で定める日」として施行日が未確定の改正が公布されることがある。複数の未施行改正が存在する場合、施行順序が不確定となり、法令の将来状態が一意に定まらない。

**調整規定（Adjustment Provisions）**: 施行順序の不確定性に対処する仕組み。改正法令内に「他の改正法令の施行状態に応じて自身の改正内容を読み替える」条項を設ける。

**割り込み施行**: 先に公布された改正法令よりも後に公布された改正法令が先に施行されるケース。タイムラインの挿入が発生する。

**2段ロケット（n段ロケット）**: 単一の改正法令が同一法令に対して複数の改正規定を持ち、それぞれ異なる施行日を持つパターン。後段の改正は前段の適用を前提とする。

### 6.3 バージョン管理

**法的タイムライン**: 施行日に基づく法令の状態遷移。`asof` パラメータでの時点指定クエリに対応。

**編集タイムライン**: 法令案の起草・修正の過程。API には expose されないが、データの更新タイミングに影響する。

**正誤**: 官報掲載後の誤記訂正。法的タイムラインとは別の編集バージョンを生成する。

> 詳細: `../docs-alpha/law-revisions.md`, `../docs-alpha/version-control.md`
