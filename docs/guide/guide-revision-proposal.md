## 4. 各項目の詳細

### R-09: Qwen3.5 thinkingモード対応の全面組み込み【最高優先度】

**現状:** ガイドはQwen3.5のthinkingモードについて一切言及していない。コード例はすべて `choices[0].message.content` をそのまま使用する前提であり、thinkingブロックの分離処理が欠落している。`max_tokens` のデフォルトは全Stage通じて1024。`-c`（コンテキスト長）のデフォルトは8192。

**問題（実測に基づく）:**

**(1) Qwen3.5はファミリー全体でthinkingがデフォルトonである。**

公式ドキュメント（HuggingFace）に「Qwen3.5 models operate in thinking mode by default」と明記。`<think>\n...\n</think>\n\n` で囲まれたthinkingコンテンツが最終応答の前に生成される。Qwen3.5はQwen3の `/think` `/nothink` ソフトスイッチを公式にはサポートしておらず、無効化には `chat_template_kwargs: {"enable_thinking": false}` のテンプレートレベル制御が必要だが、llama.cppではOSごとのJSONエスケープの差異等で設定が非自明。

Unslothドキュメントの「Small models disables by default」は実態と乖離している。4B, 9Bモデルでも実際にはthinkingブロックが生成される報告が多数あり、学習者がこの記述を信頼してthinking処理を省略すると動作しない。

**(2) max_tokens=1024ではthinkingの途中で生成が打ち切られ、結論に到達しない。**

1024トークンの大部分をthinkingブロックが消費し、最終応答が短くなるのではなく、**最終応答が存在しない**（thinkingの途中で出力が終了する）。Stage 0-Dのレイテンシ計測すら正常に動作しない致命的問題。

実測では `max_tokens=4096` に設定することで全タスクで100%のtool calling成功率を達成。ただし4096でも応答が途中で切断されるケースが会話ログ上で複数確認されている（Stage 4 Round 1 Turn 4, 5, 8等）。4096は「tool callingが動作する最低ライン」であり、「応答が途中で切れない十分なライン」ではない。thinkingブロックの長さがタスクごとに大きく変動するため、固定値での完全な制御には限界がある。

**(3) thinkingトークンはmax_tokensの生成予算を消費するが、次回リクエスト時のKVキャッシュには載らない。**

Stage 4実験レポート§4.2で確認。`reasoning_content` はAPIレスポンスに含まれるが、次のリクエストのmessagesには含まれない（dischargeされるため）。10ターン分のthinkingが蓄積すると推定~15-20Kトークンで即座にコンテキスト超過するが、実際にはそうならない。これはコンテキスト管理にとって好都合。

**(4) `-c 8192` は不必要に窮屈。**

モデルロード後のVRAM使用量が~20GB程度（Q4_K_M基準）、残り~12GB。Qwen3.5-35B-A3BのGQAによるKVキャッシュ効率は高く（64Kでも~1.2GB程度）、`-c 16384` でも余裕をもって収まる。8192では長いツール結果を含む複数ターン対話でコンテキスト不足になりうる。

**変更案:**

**(a) §0に「Qwen3.5 thinkingモードの基礎知識」セクションを新設。**

**(b) Stage 0の基盤コードにthinkingブロック分離処理を組み込む。**

```scala
case class LlmResponse(thinking: Option[String], content: String)

def separateThinking(raw: String): LlmResponse = {
  val thinkPattern = """(?s)<think>\n?(.*?)\n?</think>\n*(.*)""".r
  raw match {
    case thinkPattern(thinking, content) =>
      LlmResponse(Some(thinking.trim), content.trim)
    case other =>
      LlmResponse(None, other.trim)
  }
}
```

**(c) 全Stageの `max_tokens` デフォルトを4096に変更。**

| 箇所 | 現行値 | 変更後 |
|------|--------|--------|
| Stage 0-D レイテンシ計測 | 32 / 128 / 256 | 512 / 2048 / 4096 |
| Stage 1 構造化出力 | 512 | 4096 |
| Stage 2-3 tool calling | 1024 | 4096 |
| Stage 5 計画生成 | 1024 | 4096 |
| Stage 6 評価 | 256 | 2048 |

**(d) `-c` の推奨値を16384に変更。** VRAMバジェット根拠:

| 項目 | 消費量 |
|------|--------|
| モデル重み (UD-Q4_K_XL) | ~22 GB |
| KVキャッシュ (16K ctx, GQA) | ~0.3 GB |
| CUDAオーバーヘッド等 | ~1-2 GB |
| **合計** | **~24-25 GB** |
| **32GB VRAMに対する残余** | **~7-8 GB** |

**(e) thinkingブロックのメッセージシーケンスへの非包含を明示。** R-04（入力選択問題）との関連で、thinkingトークンが入力選択問題の範囲外であることを記載。

### R-01: GPGPU WS側モデルの変更

**現状:** `Qwen3.5-35B-A3B-Uncensored-HauhauCS-Aggressive-Q4_K_M.gguf`（~20GB）を前提としている。

**問題:**
- Uncensoredバリアントはtool callingの学習分布からずれている可能性がある
- Q4_K_Mは均一量子化であり、Unsloth Dynamic 2.0（UD-Q4_K_XL）の方が重要レイヤーをQ6_K/Q8_0にアップキャストすることで同等サイズで品質が向上する
- Unsloth最新版はtool calling改善済みの修正テンプレートを同梱しており、§0.3で述べたJinjaバグも緩和される

**変更案:** 推奨モデルを `unsloth/Qwen3.5-35B-A3B-GGUF` の `UD-Q4_K_XL`（22.2GB）に変更。

**Q6_K / Q8_0 への引き上げは非推奨。** 実測ファイルサイズに基づくアセスメント:

| 量子化 | ファイルサイズ | 32GB VRAM残余（推定） | 判定 |
|--------|-------------|---------------------|------|
| UD-Q4_K_XL | 22.2 GB | ~8 GB | ✅ 推奨 |
| UD-Q5_K_XL | 26.4 GB | ~4 GB | △ 条件付き可 |
| UD-Q6_K_S | 28.5 GB | ~2 GB | × 実用困難 |
| UD-Q6_K_XL | 32.1 GB | 0 GB | × 不可 |
| UD-Q8_K_XL | 48.7 GB | — | × 不可 |

加えて、Stage 3の実験でQ4_K_Mの時点で全タスク100%成功（天井効果）が観察されており、量子化精度を上げても品質差が検出されない可能性が高い（Stage 3レポート§4.5）。VRAMの余裕をコンテキスト長の拡張余地として保持する方が実用上の価値が大きい。

### R-02: Mac単体実行時の構成追記

**現状:** ガイドはGPGPU WS上のllama-serverのみを前提としている。

**変更案:** §0.4に以下を追記する。

**(a) 推論サーバの選択:**

| 選択肢 | tool calling | thinkingモード制御 | 推奨 |
|--------|------------|------------------|------|
| `mlx_lm.server`（Apple公式） | 不完全 | 不明 | × |
| `mlx-openai-server`（サードパーティ） | 対応（Qwen3系parser あり） | 適切に制御可能 | ○ |
| Ollama | Qwen3.5 GGUFが動作しない（mmproj問題） | `enable_thinking` が効かない報告あり | × |

mlx-openai-serverを推奨する根拠は2つ。第一にtool callingサポート。第二にthinkingモードの制御がllama.cppより適切にできる点。9Bモデルの推論品質がthinkingモードの制御に強く依存するため（下記(b)参照）、この制御の良し悪しは実験結果の再現性に直結する。

**(b) 9Bモデルのthinkingモード影響（実測データ）:**

画像判別タスク（mlx-vlm使用、特定の判別タスクを2-3バッチ処理）での観察:
- **thinking-onにすると処理時間が約20倍に増加**
- **thinking-onにすると正答率が約1/4に低下**

9Bモデルではthinkingモードが推論品質を**劣化**させる。thinkingの推論過程でモデルがdriftし、正しい方向に向かっていた判断が途中で捻じ曲げられる。thinkingを切った方が時間・品質の両面で良い。

これは35B-A3Bでの観察（thinking-onで全タスク100%成功。thinkingがtool callingの安定性に寄与している可能性）と**正反対**の結果であり、thinkingモードの有効性にはモデルサイズに依存する閾値が存在することを示唆する。

**(c) Mac単体実行時の推奨:**
- **モデル:** Qwen3.5-9B（MLX形式）、**thinking-off**で運用
- **切り替え方法:** `LLM_BASE_URL` 環境変数の変更のみ。クライアントコードは共通

```bash
# GPGPU WS使用時
LLM_BASE_URL=http://<GPGPU-WS-IP>:8080 scala-cli run agent.scala

# Mac単体時
LLM_BASE_URL=http://localhost:8000 scala-cli run agent.scala
```

- **注意事項:** 9B（thinking-off）は35B-A3B（thinking-on）よりtool calling判断品質が低い。Stage 2-3の実験結果が環境間で異なる可能性があり、これ自体がモデルサイズとtool calling品質の関係を理解する学習機会となる。

### R-03a: ツールの対応範囲の設計と拡張

**現状:** Stage 2でKnownLawsを基本六法にハードコードし、Stage 3でfind_law_by_keywordを追加しているが、両ツール間の連携（find_law_by_keywordで得た法令IDをsearch_statuteに渡す）がガイドに明示されていない。

**問題:** Stage 4 Round 1 Turn 6で、実在する法令（個人情報保護法）がKnownLawsに含まれないためにツールがネガティブ結果を返し、モデルが内部知識にフォールバックするハルシネーションが観察された。

**Stage 4 Round 2で実証済み:** Round 2ではe-Gov APIクライアントを再実装（lawIdベース設計）し、Turn 5の`find_laws`結果から`[ID: 415AC0000000057]`を取得、Turn 6でモデルが`get_article("415AC0000000057", "1")`を呼び出して正しく条文を取得した。**モデルが前のターンのツール結果からlawIdを抽出して次のツール呼び出しの引数に使うAdaptive Planningが自発的に発生した。**

**変更案:** Stage 3に以下のツール連携パターンを明示的に含める:
- find_law_by_keywordで法令名→法令IDを動的に解決
- 解決した法令IDをsearch_statuteに渡す
- KnownLawsのハードコードは「高速パス」（キャッシュ的な役割）に格下げ

このパターンはStage 5（計画と分解）の「Adaptive Planning」の具体的な動機付けにもなる。ただし、Round 2で観察された通り、モデルは明示的な計画なしでもこの連鎖を自発的に実行できる。Stage 5では「この自発的な連鎖を意図的に制御・検証する方法」として位置づける。

### R-03b: ネガティブケースのテストとハルシネーション対策

**現状:** Stage 2のテストケース#1〜#5はすべてツールが成功するシナリオのみ。

**問題:** Stage 4 Round 1 Turn 6で実測されたハルシネーションの実態:

> ツール結果: `エラー: '個人情報の保護に関する法律' は登録されていません。利用可能: 行政事件訴訟法, 民事訴訟法, 刑法, 行政手続法, 民法, 憲法`
>
> モデルの応答: `個人情報保護法第1条（目的）の条文は以下の通りです。` — エラーへの言及なし。架空の条文テキストを自信を持って生成。

3つのリスクが特定された（Stage 4レポート§3.5）:
1. **出典の偽装**: ユーザーはツール経由で取得した条文だと思うが、実際はLLMの内部知識
2. **エラーの隠蔽**: ツール失敗がユーザーに伝わらない
3. **信頼性の混同**: ツール由来の確実な情報と内部知識の不確実な情報が区別できない

**変更案:**

(a) Stage 2のテストケースに以下を追加:

| # | 質問 | 種別 | 期待される動作 |
|---|------|------|------------|
| 6 | 「商法第999条の条文を教えてください」 | 存在しない条文 | 「見つかりません」と伝える |
| 7 | 「個人情報保護法の条文を見せてください」 | KnownLawsに未登録の実在法令 | 「登録されていません」と伝える |

(b) system promptへの制約を第一の対策として実装・検証: 「ツールが見つからないと返した場合は、その旨をそのまま伝え、自身の知識で補完してはならない」

(c) テストケース#7はR-03aの適用前後で2回実行:
- R-03a適用前: ハルシネーションが観察されることを確認
- R-03a適用後（Stage 3）: find_law_by_keyword連携により正常に条文を取得できることを確認

### R-04: Stage 4のアーキテクチャ概念の修正

**現状:** 「コンテキストウィンドウの有限性がアーキテクチャを規定する」と記載しつつ、`ConversationState` の実装がKVキャッシュの管理と混同されうる記述になっている。

**問題:** llama-serverの `/v1/chat/completions` はステートレスAPIであり、KVキャッシュはリクエストごとに再構築される。Stage 4の問題の本質は「KVキャッシュの管理」ではなく、**リクエストごとに送るメッセージ配列が上限トークン数に収まらなくなること**。

**変更案:** Stage 4の冒頭に以下の3層構造を明記する。

1. **永続化層**（JSONファイル/SQLite）: 全会話履歴を保存する。失われない。
2. **選択層**（設計の核）: 永続化された履歴から、次のリクエストに含める部分を選択する。
3. **トランスポート層**（HTTPリクエスト）: 選択されたメッセージをJSON配列としてAPIに送る。

**R-09との関連:** thinkingトークンは生成時にGPU計算時間を消費するが、選択層の対象にはならない（dischargeされるため）。thinkingは「入力選択問題」の範囲外であり、別軸のコスト要因（推論時間・max_tokens予算消費）として位置づける。Stage 4実験レポート§4.2で「10ターン分のthinkingが蓄積すると~15-20Kトークンで即座にコンテキスト超過するが、実際にはそうならない」ことが確認済み。

### R-05: 要約ベース圧縮の段階的実装（Stage 4）

**現状:** 単純切り捨てのみを実装し、要約は「設計の検討のみ」として触れている。

**変更案:** Stage 4の実装を以下の4ステップに再構成する。

- **Step 1:** 単純切り捨てを実装し、退行を観察する。10ターン対話後、切り捨てにより文脈が失われていることを確認する（例: 「さっき調べた条文をもう一度見せて」と言ったときにモデルが文脈を失っている）。
- **Step 2:** 要約ベースの圧縮を実装する。古いメッセージ群をLLMに要約させ、system promptに「これまでの対話の要約」として挿入する。要約設計はR-06の原則に従う。
- **Step 3:** 同じ10ターン対話を再実行し、Step 1との品質差を比較する。
- **Step 4:** 要約で失われた情報を特定する。基盤Cの「矮小化」概念との接続点。

### R-06: ツール取得データの要約除外原則（Stage 4）

**現状:** 要約の設計方針について具体的な指針がない。

**問題:** 要約にツール取得データ（条文テキスト等）が含まれると、モデルはそれを「既知情報」として扱い、再度問われたときにツールを呼ばずに不完全な要約から回答する。この場合、要約は切り捨てよりもむしろ品質が劣化する。切り捨てなら情報が完全に不在なのでモデルはツールを呼ぶ。

**設計原則: 再取得可能な情報は保持しない。再取得不可能な文脈のみ保持する。**

情報の構造的不在に依存する方式を採用する。「要約なので再調査せよ」とプロンプト指示で制御する方式はモデルの指示追従性に依存し、動作の予測可能性が低い。ツール呼び出し判断はStage 2-3で検証済みの「コンテキストに情報がないときにツールを呼ぶ」挙動に帰着させる。

要約プロンプトの設計:

- **対話の文脈**（残す）: ユーザの目的、議論のトピック、到達した結論
- **ツール取得データ**（残さない）: 条文テキスト、検索結果の具体的な内容

生成される要約の例:
```
これまでの対話の要約:
- ユーザは不法行為に基づく損害賠償請求の要件を調査している
- 民法709条の条文をsearch_statuteで取得し、内容を確認した
- 条文から4つの成立要件を特定し、各要件の意味を議論した
- 次に因果関係の要件について判例の検討に進む予定だった
```

### R-07: ツールネガティブ結果の整合性チェック（Stage 6）

**現状:** Stage 6の評価基準は `source_cited`、`internally_consistent`、`answers_question` の3項目。

**問題:** `source_cited` は「条文番号が応答中に含まれているか」を見ているだけであり、モデルが架空の条文番号を自信を持って提示した場合は `true` と判定されてしまう。R-03bのTurn 6事例がまさにこれに該当する。

**変更案:** 以下の機械検証項目を追加:

- **`tool_result_consistent`**: ツールがネガティブ結果を返した場合に、最終回答がそのネガティブ結果と整合しているか。ツールが失敗したのに回答に当該条文の具体的な内容が含まれていれば `false`。

### R-08: Stage 7の全面差し替え — thinkingブロック分析

**現行:** Stage 7は「研究タスクへの接続」としてC-ε（Agentic Legal Research）等の研究タスクを列挙。

**問題:** 研究タスクはそれ自体で独立したプロジェクトに切り出すべきであり、学習ガイドのスコープに含めるべきではない。学習の連続性が断絶している。

**変更案:** Stage 7を「thinkingブロック分析」に変更する。

**(a) 位置づけ:** Stage 0-4で生成され続けていたがdischargeしていたthinkingブロックを、このステージで初めて分析対象にする。「構築→高次設計→観察→統合」の「観察」フェーズ。thinkingは「新たに有効化する機能」ではなく「最初からずっと出力されていたが捨てていたデータ」。

**(b) 観察の焦点（3つに限定）:**

- **ツール呼び出し判断の過程:** 「ツールを呼ぶ」と決定するまでにthinking内で何が起きているか
- **ネガティブ結果後のハルシネーション過程（R-03b関連）:** ツールが「見つかりません」を返した後、thinking内でモデルが内部知識へのフォールバックを判断する瞬間の可視化
- **ツール選択（ルーティング）の推論過程:** Stage 3のdescriptionによる精度差がthinking内でどう現れるか

**(c) thinkingブロックのコスト計測:** thinkingトークン数 vs 最終応答トークン数の比率をtool callingラウンドごとに記録する。

**(d) 35B-A3B vs 9Bの比較:** 35B-A3Bのthinkingブロックでは「考えを深めて正しい結論に至る過程」が観察でき、9Bでは「driftして結論が捻じ曲げられる過程」が観察できるはず。両者の比較はモデルサイズとthinking品質の関係を理解するうえで有益。

**(e) 成果物:** 観察レポートではなく、**thinkingブロック分析に基づくsystem prompt / tool description改善の提案とその検証**。Stage 2-6の設計にフィードバックするループを閉じる。

**(f) CC分析の二重性への注意:** CCにthinkingブロックを分析させる場合、生のthinkingテキストを自分の目でも読む作業を省略せず、CCの分析と自分の観察の一致・不一致を記録する手順を明示。

**(g) 研究タスクの扱い:** 現行Stage 7のC-ε, D-δ, E-β等は「付録: 研究タスクへの発展」に移動し、学習ガイドのStage構成からは切り離す。

### R-10: 期待される失敗パターンの記述調整

**現状:** ガイドは「ツールを呼ぶべき質問でテキスト回答が返る」「tool callのJSONが壊れる」「ルーティングの誤り」といった失敗を予想している。

**問題:** Stage 1-4の全実験でほぼ100%の成功率。天井効果が一貫して観察されている（Stage 3レポート§4.2）。

**変更案:**

(a) 「こういう失敗が起こるはず」という断定的記述を、「起こりうる失敗パターン」として提示し、以下の注記を加える:
- 実際に発生するかはモデルの能力・サイズ・量子化・thinkingモードの有無に依存する
- 35B-A3B（thinking-on）で発生しなかったパターンが、9Bモデルでは発生する可能性がある
- 発生しなかった場合でも、パターンを知っておくことには防御的設計の根拠としての価値がある

(b) Stage 3 §4.3で指摘された「失敗が顕在化する条件」を具体的なテストケースとしてガイドに追加:
- ツール間の境界が曖昧なケース（例: 「民法の構成は？」→ searchStatute? findLawByKeyword?）
- 温度を上げた場合の出力バリエーション

### R-11: calculate実装のJVM互換性修正

**現状:** ガイドのcalculate実装は `javax.script.ScriptEngineManager` を使用している。

### R-13: Stage 8の新設 — REPL実装と統合【新規】

**位置づけ:** Stage 0-7で個別に構築・検証した要素を、一貫したインタラクティブシステムとして統合する最終ステージ。学習カリキュラムの最終成果物であると同時に、研究タスクに実際に使い始められるツールとなる。

**Stage 5（REPL）ではなくStage 8とする根拠:**

REPLをStage 4の直後（旧Stage 5の位置）に置くと、学習者は自由対話の中で人間が手動で計画の分解・評価の代行・thinkingの観察をすることになり、Stage 5-7の学習内容を「それと意識することなく実質的に実現してしまう」。後続Stageが「既に体験的に知っていることの形式化」になり、発見の体験が薄れる。

Stage 8に置けば、Stage 5（計画）、Stage 6（自己評価）、Stage 7（thinking分析）で学んだ要素をREPLのどこにどう組み込むかという**統合設計**の問題になる。これは新しい設計判断を要求するStageであり、単なるstdinラッパーにはならない。

**学習目標:** Stage 0-7で個別に構築・検証した要素を、一貫したインタラクティブシステムとして統合する。各要素間の相互作用に関する設計判断を行い、実用可能な法学ドメイン向けagentic対話システムを完成させる。

**実装上の設計判断（Stage 5-7の統合）:**

- ユーザの入力が単純な質問か複合タスクかによって、Stage 5のplanning層を介入させるかどうか
- 応答に対してStage 6のself-evaluationループを自動実行するかどうか、実行する場合のトリガー条件
- Stage 7のthinkingブロックを表示するか・要約するか・隠すかの制御をどうUIに露出するか
- セッション管理（`/save`, `/load`, `/clear` 等のメタコマンド）
- ツール呼び出しの経過表示（呼び出し中のツール名と引数のリアルタイム表示）

**成果物:** 動作するREPLアプリケーション。付録に移動した研究タスク（C-ε, D-δ, E-β等）に着手する際の出発点。
