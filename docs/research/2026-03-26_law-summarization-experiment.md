# 法令全文要約実験

> 実施日: 2026-03-26
> 環境: GPU WS (RTX PRO 4500 Blackwell 32GB), llama.cpp build 8462, `-c 262144`
> モデル: Qwen3.5-35B-A3B-UD-Q4_K_XL.gguf (Unsloth, 20.70 GiB)
> 対象法令: 民法（明治二十九年法律第八十九号）
> インフラ特性: `2026-03-26_qwen35-35b-kv-cache-analysis.md` を参照

## 1. 実験目的

法令全文の LLM 投入による要約が実用的かどうかを検証する。KV キャッシュ分析でコンテキスト制約の消失が確認されたことを受け、2つのアプローチを比較:

1. **全文一発**: 法令全文を1回のリクエストで要約
2. **階層分割（Map-Reduce）**: 編単位で個別に要約 → 統合要約

背景: Stage 8 テストで判明した「196条ハルシネーション事件」は、条文内容検索ツールの不在が構造的原因。`search_within_law` ツールは実装済みだが、法令全体の「見取り図」を得る機能（`get_law_summary`）の設計に本実験の結果が直結する。

## 2. 対象法令の構造

民法は日本の私法の基本法で、五編構成。実験対象として「最大級の法令」を選択。

| パート | Sentence chars | 推定 tokens | 実測 prompt tokens |
|--------|---------------|-------------|-------------------|
| 第一編 総則 | 16,218 | ~6K | 11,050 |
| 第二編 物権 | 32,521 | ~13K | 22,519 |
| 第三編 債権 | 57,025 | ~23K | 38,535 |
| 第四編 親族 | 21,979 | ~9K | 15,924 |
| 第五編 相続 | 24,162 | ~10K | 17,036 |
| 附則（65 sections） | 36,599 | ~14K | 25,768 |
| **全体** | **213,435** | **~71K** | **122,559** |

推定（chars/3）と実測の乖離: 71K vs 122K tokens。日本語法令テキストのトークン化は ~1.7 chars/token（3 chars/token の推定より効率が悪い）。system prompt + 構造マークアップのオーバーヘッドも含む。

## 3. 実験1: 全文一発要約

### 手法

- Sentence + ArticleCaption + ArticleTitle を抽出、構造マークアップ付きテキスト化
- System prompt で6要件（目的・構造・定義・中核規律・罰則・附則）を指定
- `max_tokens=8192`, `temperature=0.0`

### 結果（実機確認）

| 指標 | 値 |
|------|-----|
| 入力テキスト | 213K chars |
| プロンプトトークン | 122,559 |
| 生成トークン | 3,831 |
| 合計 | 126,390 / 262,144 (**48%**) |
| 所要時間 | **81s** |
| PP 速度 | 2,916 tok/s（単独実行） |
| TG 速度 | 99 tok/s |
| 出力 | 2,969 chars |

### 品質評価

| 要件 | 充足度 | 備考 |
|------|--------|------|
| 目的と規律対象 | 良好 | 第1条・第2条を正しく引用 |
| 全体構造 | 良好 | 五編の構成と各部分の役割を記述 |
| 定義規定 | 良好 | 人・法人・物・法律行為・時効・婚姻・相続 |
| 中核規律 | 良好 | 行為能力、物権変動、契約自由、不法行為、相続分 |
| 罰則 | 良好 | 民事制裁中心、過料規定の存在を指摘 |
| **条番号正確性** | **1件誤り** | 不法行為を「第七百十九条」と記載（正: 第七百九条） |

## 4. 実験2: 階層分割（Map-Reduce）要約

### 手法

- **Phase 1**: 5編 + 附則の6パートに分割、4並列（`n_parallel=4`）で各編を個別要約
- **Phase 2**: 6編の要約を統合し、全体要約を生成
- `max_tokens=4096`（Phase 1）, `max_tokens=8192`（Phase 2）, `temperature=0.0`

### Phase 1 結果（実機確認、llama-server ログ）

| パート | Slot | 時間 | Prompt tok | Gen tok | PP (tok/s) | TG (tok/s) | 出力 |
|--------|------|------|-----------|---------|-----------|-----------|------|
| 総則 | 2 | 135s | 11,050 | 4,096 | 388 | 39 | **0 chars** |
| 物権 | 0 | 112s | 22,519 | 3,530 | 1,722 | 36 | 2,826 chars |
| 債権 | 3 | 125s | 38,535 | 4,096 | 1,671 | 40 | 1,631 chars |
| 親族 | 1 | 123s | 15,924 | 4,096 | 1,283 | 37 | 698 chars |
| 相続 | 0 | 84s | 17,036 | 4,096 | 1,761 | 55 | **0 chars** |
| 附則 | 1 | 68s | 25,768 | 3,948 | 1,738 | 74 | 2,750 chars |

**content=0 問題**: 総則・相続で `completion_tokens=4096`（上限ヒット）にもかかわらず出力が空。Thinking ブロックが `max_tokens` を全消費したと推定。→ `max_tokens=8192/16384` で再実験が必要。

**スロットスケジューリング**:

```
[Slot 0] ─── 物権(112s) ──── 相続(84s) ───
[Slot 1] ─── 親族(123s) ──── 附則(68s) ───
[Slot 2] ─── 総則(135s) ───────────────── Phase2(49s) ──
[Slot 3] ─── 債権(125s) ─────────────────

→ 前半: 4並列（~125s）→ 後半: 2並列（~85s）→ Phase 2: 単独（49s）
```

### Phase 2 結果（実機確認）

| 指標 | 値 |
|------|-----|
| 入力（6編の要約結合） | 8,014 chars |
| Prompt tokens | 5,619 |
| Completion tokens | 4,907 |
| PP 速度 | 2,350 tok/s（単独実行） |
| TG 速度 | 105 tok/s |
| 時間 | 49.2s |
| 出力 | 3,423 chars |

### 全文一発 vs 階層分割 比較

| 指標 | 全文一発 | 階層分割 |
|------|---------|---------|
| Wall time | **81s** | 254s |
| Prompt tokens | 122,559 | 136,451 |
| Completion tokens | 3,831 | 28,769 |
| 合計 tokens | 126,390 | 165,220 |
| 条番号正確性 | 1件誤り（709→719） | **正確** |
| 附則・経過措置 | 概要のみ | **具体的な改正法・条文番号付き** |
| 構造の解像度 | 編レベル | **章・節レベル** |

### 階層分割の品質優位性

Phase 2 の統合要約は以下の点で全文一発より優れていた:
- **条番号の正確性が向上**: 709→719 の誤りが解消
- **附則・経過措置が具体的**: 平成29年改正（債権法）、平成30年改正（成年年齢）等の個別経過措置まで記述
- **構造の解像度**: 編内の章・節レベルの規定が具体的に言及

推定原因: 各編を個別に処理することで、LLM のアテンション予算が分散せず、各パートにより多くの注意を払える。

## 5. 性能分析

### スロット競合の影響（詳細はインフラ分析を参照）

| 条件 | PP (tok/s) | TG (tok/s) |
|------|-----------|-----------|
| 単独実行 | 2,916 | 99 |
| 4並列 | 388-1,722 | 36-40 |
| 2並列 | 1,738-1,761 | 55-74 |

4並列の PP 劣化が激しい（最悪 7.5 倍）。Phase 1 の wall time が 254s のうち ~135s はスロット競合による速度低下が原因。

### GPU サーマル特性（詳細はインフラ分析を参照）

- 全文一発: 49→53°C（+4°C）
- 階層分割 Phase 1: 50→73°C（+23°C、ピーク）
- サーマルスロットリングなし（ピーク 73°C < 閾値 83-90°C）
- TDP 200W に張り付き（power-limited）

## 6. 設計指針

### アプローチ選択

| 用途 | 推奨 | 理由 |
|------|------|------|
| インタラクティブ（REPL） | 全文一発 | 81s で実用的なレイテンシ |
| 品質重視（報告書作成等） | 階層分割 | 条番号正確性・附則詳細度が高い |
| 中小規模法令 | 全文一発 | コンテキストに余裕、速度優位 |
| 大規模法令（民法級） | 要検証 | 全文一発で 48% 消費。品質差次第で判断 |

### `get_law_summary` ツール設計への反映

- **全文一発を基本実装とし、階層分割をオプションに**
- 法令サイズに応じた自動選択: prompt tokens < 50K → 全文一発、≥ 50K → 階層分割
- `max_tokens`: 8192 以上必須（Thinking 消費対策）
- 並列度: Phase 1 は 2-3 並列が最適（4並列は PP 劣化が大きすぎる）

## 7. 実験の結論と Deferred 課題

### Round 1 の結論（Stage EX scala-cli 実験で確定）

- **max_tokens=8192 で content=0 問題は解消**（C4: 0/6, C2: 0/6 パート欠損）
- **2並列×3バッチ（C2）がパレート最適**: 248s, 品質完全
- **3並列以上は品質劣化**: content=0 + TG 速度低下
- **日本語法令テキストのトークン化**: ~1.7 chars/token（一般的推定 3 chars/token は法令テキストに不適）

### Deferred 課題（ツール実装フェーズで段階的に検証）

以下は検証価値があるが、現時点の机上実験としては ROI が低いと判断した課題。ツール実装後の実運用フィードバックに基づいて段階的に検証する。

| 課題 | 意義 | Defer 理由 |
|------|------|-----------|
| System prompt メタデータ付与 | Thinking 探索空間の縮小 → content=0 軽減 | `get_law_summary` 実装に組み込み、実運用で検証 |
| KV キャッシュ量子化 | 並列時帯域圧力の軽減 | サーバ全体に影響。品質劣化検証コスト大。性能問題顕在化後に検証 |
| キャッシュ無効化での PP 再計測 | PP 速度の正確な並列度依存性 | C2 最適の結論は wall time+TG で有効。PP 絶対値が設計を変える見込み低 |
| 分割戦略の最適化 | 意味論的分割、LLM 自律分割 | 探索空間が広い。編単位で実用品質確認済み |
| 中小規模法令テスト | 行政手続法等での速度・品質 | ツール実装後に自然に得られるデータ |

詳細は `stages/stage-ex/summarization/RESULTS.md` §4.6-4.7 を参照。

## 8. Addendum: プロンプトキャッシュの交絡（事後発見）

scala-cli 実験（C4→C2→C3→C5）をサーバ再起動なしで連続実行したため、llama-server のプロンプトキャッシュが実験間で共有されていた。

**影響**:
- PP 速度の条件間比較にキャッシュヒット/ミスの交絡がある
- Wall time と TG 速度の比較は有効（キャッシュと独立）
- C2 がパレート最適という結論は wall time + 品質ベースで維持される

**SWA キャッシュ破損**: C5 でプロンプトキャッシュが上限（8,192 MiB）の 90% に到達し、Hybrid Attention-SSM の SWA との非互換でチェックポイント破損が発生。一部タスクが全再処理に回った。

**推奨**: 再実験時はサーバ再起動 or `--cache-ram 0` でキャッシュを無効化し、各条件を独立実行すべき。詳細は `stages/stage-ex/summarization/NOTES.md` の「llama-server ログ分析による再評価」セクション。
