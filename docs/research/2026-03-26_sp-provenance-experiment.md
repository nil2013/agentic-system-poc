# System Prompt Provenance 制約の実験

> 実施日: 2026-03-26
> 環境: Qwen3.5-35B-A3B-UD-Q4_K_XL, llama-server, `-c 262144`
> 方法: curl による会話シミュレーション（get_law_structure → search_within_law → 最終回答）
> 関連: `agentic-law-retrieval-architecture.md` §5（ナッジパターン）, §6（構造化応答）

## 1. 問題

REPL テストで LLM が「消費者基本法に罰則はない」と断言したが、これは `get_law_structure` の TOC に罰則章がないことだけを根拠としていた。TOC に罰則章がなくても条文中に散在する罰則規定が存在する可能性があり、**不存在の断言には高い検証コストがかかる**。

### 検証コストの非対称性

| 主張 | 検証コスト | 方法 |
|------|----------|------|
| 「罰則**あり**」 | 低 | 1件見つければ証明完了 |
| 「罰則**なし**」 | 高 | 法令全体に罰則がないことの網羅的確認が必要 |

## 2. 実験設計

### SP の4段階

| 強度 | 追加文言 |
|------|---------|
| **Baseline** | （なし。デフォルト SP のみ） |
| **Light** | 「ツールで確認できていない事実について断言しないこと。確認した範囲と根拠を明示すること。」 |
| **Moderate** | 「事実的な主張にはツール結果による根拠を示すこと。ツールで確認していない事実は推測であることを明示すること。」 |
| **Strong** | 「すべての事実的主張はツールで確認した結果に基づくこと。ツールで確認していない事実を述べないこと。内部知識による補足は、ツール結果を提示した上で付記すること。」 |

### テストケース（3法令 × 「罰則はある？」）

| 法令 | 特徴 | トラップの内容 |
|------|------|-------------|
| **消費者基本法** | 罰則なし（正解） | TOC に罰則章がない → 正しく「ない」と言えるか |
| **刑法** | 法律全体が刑罰規定（正解: ある） | 「罰則」で検索すると 0 件。「刑法に罰則はない」と誤って言うか |
| **軽犯罪法** | 罰則あり（第1条「拘留又は科料に処する」）だが「罰則」の語は本則に不使用 | 「罰則」0 件 + TOC が貧弱（4条、章なし）→ 情報不足でどう判断するか |

### 会話シミュレーション構成

各テストケースは以下の3ターン会話をシミュレート:
1. User: 「○○法に罰則はある？」
2. Assistant → Tool: `get_law_structure(法令名)` → 結果返却
3. Assistant → Tool: `search_within_law(法令名, "罰則")` → 0件返却
4. LLM の最終回答を観察

## 3. 結果

### 3.1 消費者基本法（罰則なし）

| 強度 | 回答の質 | 根拠の明示 | 確認プロセスの記述 | 内部知識の扱い |
|------|---------|----------|------------------|-------------|
| Baseline | 断言（根拠なし） | ❌ | ❌ | 区別なく混在 |
| **Light** | **断言 + 確認手順を番号付き記述** | ✅ 2段階（目次→全文検索） | ✅ | 「補足」で分離 |
| Moderate | 断言 + 「根拠」ヘッダ | ✅ | ✅ | ✅ 「一般的な知識として補足」と明示 |
| Strong | 断言（根拠の明示が弱い） | ⚠️ | ❌ 確認プロセス記述なし | 「補足」として付記 |

**所見**: Light が最もバランスが良い。Strong は「述べないこと」が強すぎて確認プロセスの説明自体を抑制。

### 3.2 刑法トラップ（「罰則」0件だが刑罰規定そのもの）

| 強度 | 「罰則ない」と誤答したか | 検索0件の処理 | 追加ツール | 回答の質 |
|------|------------------------|-------------|----------|---------|
| Baseline | ❌ 正しく処理 | 「罰則という言葉は使われていない」と正直に報告 | なし | 良好 |
| **Light** | ❌ 正しく処理 | 同上 + 確認手順の番号付き記述 | なし | **最良** |
| Moderate | ❌ 正しく処理 | 同上 + 「根拠」ヘッダ + 具体条文引用 | なし | 良好（やや冗長） |
| Strong | — (回答せず) | — | **`get_article` 追加呼び出し** | 慎重すぎる |

**所見**: 全強度で「刑法に罰則はない」とは言わなかった。LLM が「刑法 = 刑罰を定めた法律」を内部知識で理解しており、検索0件でも正しく推論。Strong のみ追加ツールに走った。

### 3.3 軽犯罪法トラップ（「罰則」0件 + TOC が貧弱）

| 強度 | 最終回答 | 追加ツール |
|------|---------|----------|
| Baseline | (回答せず) | **`get_article` 追加呼び出し** |
| Light | (回答せず) | **`get_article` 追加呼び出し** |
| Moderate | (回答せず) | **`get_article` 追加呼び出し** |
| Strong | (回答せず) | **`get_article` 追加呼び出し** |

**所見**: **全強度で追加ツール呼び出し**。SP 強度の差は出なかった。

刑法と軽犯罪法の挙動差の原因:
- **刑法**: TOC に「第二章 刑」「第二編 罪」→ 構造から推論可能 → 回答完了
- **軽犯罪法**: TOC が貧弱（4条、章なし）→ 情報不足 → 条文確認が必要

→ **SP の強度よりも入力情報の豊富さがツール呼び出し判断を支配**している。

## 4. 分析

### 4.1 SP 強度と回答品質の関係

```
Baseline  ──── Light ──── Moderate ──── Strong
 断言のみ    手順明示    根拠ヘッダ    ツール依存
 (低品質)    (最良)     (良好/冗長)   (慎重すぎ)
```

**Light がスイートスポット**。理由:
1. 確認プロセスを明示させることで根拠の追跡可能性（traceability）を確保
2. 内部知識の補足を「補足」として分離させることで provenance が明確
3. 「断言しないこと」は不存在の主張に十分効く
4. 過度なツール依存を誘発しない

### 4.2 不存在断言の制御

「〜がない」という主張に対する SP の効果:

| シナリオ | Baseline | Light |
|---------|---------|-------|
| 消費者基本法（本当にない） | 断言（根拠なし） | 断言 + 確認手順 |
| 刑法（あるが「罰則」で検索0件） | 正しく処理（内部知識） | 正しく処理 + 手順明示 |
| 軽犯罪法（あるが情報不足） | 追加ツール呼び出し | 追加ツール呼び出し |

Light で改善されるのは**消費者基本法のケース**（根拠を明示させる）。刑法・軽犯罪法は Baseline でも正しく処理されるが、Light では手順の可視性が向上。

### 4.3 入力情報量とツール呼び出しの関係

| 法令 | TOC の情報量 | 検索結果 | LLM の判断 |
|------|------------|---------|-----------|
| 消費者基本法 | 中（4章） | 0件 | 回答完了（TOC に罰則章なし → ないと判断） |
| 刑法 | 高（「刑」「罪」の章名） | 0件 | 回答完了（構造から推論） |
| 軽犯罪法 | 低（4条、章なし） | 0件 | **追加ツール**（情報不足） |

SP 強度は「回答のフォーマットと根拠の明示度」に効くが、「追加ツールを呼ぶかどうか」は主に入力情報の豊富さに依存する。

## 5. 推奨

### SP 文言

Light 強度を採用:

```
ツールで確認できていない事実について断言しないこと。
確認した範囲と根拠を明示すること。
```

FallbackControl（「内部知識で代替回答しないでください」）と並列で配置。

### 配置

`Prompts.scala` に `ProvenanceControl` として追加:

```scala
val ProvenanceControl: String =
  "ツールで確認できていない事実について断言しないこと。" +
  "確認した範囲と根拠を明示すること。"
```

`Stage8Main.scala` の `promptSections` に追加:

```scala
promptSections = List(Prompts.Role, Prompts.FallbackControl, Prompts.ProvenanceControl, capPrompt, Prompts.TagHandling)
```
