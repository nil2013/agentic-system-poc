# Stage 実行プロトコル

各ステージで統一的に適用する手順。結果の比較可能性と再現性を担保する。

---

## 1. 準備フェーズ

### 1.1 ガイド確認

- `docs/guide/agentic-system-learning-guide-scala.md` の該当セクションを通読する
- 予定作業・仮説（ガイドの予想）・成果物を把握する

### 1.2 ディレクトリ作成

```
stages/stageN/
```

### 1.3 RESULTS.md 骨格作成

IMRaD 形式で骨格を作成する。**実験実行前に** Introduction と Methods を記述する。

```markdown
# Stage N: [タイトル] — 実験レポート

> 実施日: YYYY-MM-DD
> 環境: [端末] → [推論サーバー構成]
> モデル: [モデル名・量子化]
> ガイド参照: `docs/guide/agentic-system-learning-guide-scala.md` §X.X–§X.X
> 実験ノート: [`NOTES.md`](NOTES.md)

## 1. Introduction
### 1.1 目的
### 1.2 予定していた作業（ガイドの該当セクションへのポインタ付き）
### 1.3 仮説（ガイドの予想）

## 2. Methods
### 2.1 題材 / 入力データ
### 2.2 実験設計（パラメータ、試行回数、制御変数）
### 2.3 評価指標
### 2.4 実装（スクリプトファイル名）

## 3. Results
### 3.1 全体サマリー
### 3.2 詳細結果
### 3.3 会話品質の評価（会話ログがある場合）
### 3.4 遭遇した問題

## 4. Discussion
### 4.1 ガイドの予想との比較
### 4.2 会話ログに基づく定性的分析
### 4.3 次ステージへの示唆
### 4.4 追加検証の候補（未実施）

## 5. 成果物（ファイル一覧）
## 6. 次のステップ
```

### 1.4 NOTES.md 作成

実験ノート。時系列で観察・気付きを記録する。RESULTS.md からポインタで参照。

```markdown
# Stage N 実験ノート

実験中の観察・気付きを時系列で記録。最終的に RESULTS.md に統合する。

## YYYY-MM-DD HH:MM — [観察タイトル]
[内容]
```

---

## 2. 実装フェーズ

### 2.1 スクリプト作成

- `stages/stageN/` に scala-cli スクリプトを配置
- 端末依存の値は環境変数で参照（`sys.env.getOrElse("LLM_BASE_URL", ...)`）
- ガイドのサンプルコードは参考にするが、ガイドから逸脱する場合は NOTES.md に理由を記録

### 2.2 ガイドからの逸脱ポリシー

- ガイドの実験設計に **明らかな欠陥がない限り、逸脱しない**
- 逸脱が必要な場合は NOTES.md に理由を記録し、RESULTS.md の Methods に明記する
- 環境差異（thinking mode、サーバーパラメータ等）による調整は逸脱ではなく適応

---

## 3. 実行フェーズ

### 3.1 実験実行

- NOTES.md に実行中の観察をリアルタイムで記録する
- 問題が発生したら NOTES.md に記録してから対処する
- デバッグ過程も記録に値する（同じ問題の再発防止）

### 3.2 Ground Truth の早期検証

- 最初の数試行で ground truth とモデル出力をサンプル目視確認する
- 評価基準のバグを早期に発見する（Stage 1 の caption 括弧問題の教訓）

---

## 4. 記録フェーズ

### 4.1 RESULTS.md 完成

- Results セクションにデータを記入
- Discussion でガイドの予想との比較を記述
- 追加検証の候補を挙げる（実施するかは別判断）

### 4.1.1 会話ログに基づく定性的分析（Stage 4+）

Stage 4 以降は `ConversationLogger` により会話ログ（`conversation-log.md`）が生成される。Discussion では以下の観点で定性的分析を行う:

- **回答品質**: ツール結果を踏まえた回答の正確性・構成力
- **文脈参照**: 過去のターンの情報を正しく参照できているか
- **ツールエラー時の動作**: エラーを伝えるか、内部知識でフォールバックするか（「静かなフォールバック」問題）
- **出典の明確性**: ツール由来の情報と内部知識の区別がユーザーに伝わるか
- **特筆すべきターン**: 成功例・失敗例を会話ログから引用して記録

### 4.2 ガイドの修正必要性確認

以下の観点でガイドの修正が必要か確認する:

- **修正すべき**: 設計上のバグ（ground truth の誤り、コマンド構文の誤り等）、環境前提の誤り
- **修正しない**: 実験結果と予想の乖離（ガイドは学習カリキュラムであり、予想は学習のための仮説）
- 修正した場合はガイド末尾の修正履歴テーブルに記録する

### 4.3 コミット

- 各ステージ完了時に必ずコミットする
- ガイドの修正がある場合は別コミットにする（ステージの成果物とガイド修正を分離）

---

## 5. サンプルサイズと再実行

- full retry（ground truth 修正等でスクリプトを変更して再実行）した場合、修正箇所以外のデータは両ラウンドを合算してサンプルサイズを報告できる
- RESULTS.md の Methods にサンプルサイズの計算根拠を明記する
