### 重要な観察

#### 1. content=0 問題の完全解消

全パートで `finish_reason=stop`。max_tokens=8192 により、Thinking が消費しても content 出力の余地が十分確保された。予備実験（max_tokens=4096）では 6 パート中 2 パートで content=0 が発生していた。

#### 2. Thinking の token 消費

completion_tokens（Thinking + Content 合計）と output_chars から推定:
- completion tokens: 4,037-5,695（平均 ~4,724）
- output chars: 2,503-3,484（推定 ~1,500-2,000 tokens）
- → **Thinking が completion の 55-70% を消費**

max_tokens=4096 では Thinking だけで上限に達しうる。8192 で安全マージンが確保される。

#### 3. PP 速度の prompt サイズ依存性

| Prompt tokens | PP (tok/s) |
|---------------|-----------|
| 11,232 | 2,393 |
| 16,146 | 2,409 |
| 17,163 | 2,278 |
| 22,767 | 2,356 |
| 24,691 | 2,244 |
| 39,194 | 2,335 |

prompt サイズが 11K→39K に増加しても PP 速度はほぼ一定（2,244-2,409）。Flash Attention の効果が出ていると考えられる。

#### 4. TG 速度の安定性

101-105 tok/s で全パート均一。Autoregressive decoding はバッチサイズ=1 なので、prompt サイズや context 長にほぼ依存しない。

#### 5. 各パートの処理時間構成

各パートの wall time は prompt サイズよりも **completion tokens 数** に支配される。
- 最短: 物権（50.1s, comp=4,127）
- 最長: 附則（67.8s, comp=5,695）
- 差分: 17.7s ≈ (5,695-4,127) / 104 tok/s ≈ 15.1s → TG 時間の差とほぼ一致

PP は prompt サイズに比例するが、最大（債権 39K）でも ~17s。TG（~40-56s）が支配的。

#### 6. Thinking ブロックの内容（簡易分析）

62KB の Thinking ログを確認。各パートで:
- 要件の分析（Analyze the Request）
- 入力テキストの構造把握（Analyze the Input Text）
- 自己修正（Self-Correction/Verification）: 「条番号を推測せず、提示テキストに基づいて引用する」
- セクション別の下書き → 最終出力

「条番号は正確に記載」の指示がThinking 内で明示的に反復されており、prompt 設計が機能している。

### 重要な観察

#### 1. PP 劣化が異常に大きい

2並列で PP が **16倍劣化**（2,335 → 142 tok/s）。予備実験の 4 並列最悪値（388 tok/s）より大幅に悪い。

**仮説**: max_tokens=4096→8192 の変更により、各スロットの KV キャッシュ使用量が増加し、メモリ帯域の競合が激化した可能性。あるいは Thinking ブロック生成中の KV キャッシュ更新パターンが帯域を消費している。

予備実験（max_tokens=4096）の PP:
- 4並列最悪: 388 tok/s
- 4並列平均: ~1,430 tok/s

今回（max_tokens=8192）の PP:
- 2並列: 123-159 tok/s（全バッチで一貫して低い）

PP 劣化は max_tokens 拡大と相関している可能性あり。→ 要検証: max_tokens=4096 での C2 再実行

#### 2. Wall time は確実に短縮

PP が劣化しても、2パートを並列処理することで wall time は着実に短縮:
- Batch 0: max(61.6, 65.5) = 65.5s vs C4 逐次: 54.0+50.1 = 104.1s
- 並列効率: 104.1/65.5 = **1.59x**（理想2x に対して 79%）

#### 3. TG 劣化は軽微

TG は 103→83 tok/s（1.24倍劣化）。PP の 16 倍劣化と対照的。これは TG が autoregressive（バッチサイズ=1）で、メモリ帯域よりも計算レイテンシに支配されるため。

#### 4. Phase 2 は単独実行と同等

Phase 2 は全バッチ完了後に単独で実行されるため、PP/TG 共に C4 と同等。これは想定通り。

#### 5. バッチ間アイドル

nvidia-smi データから、Batch 0→1 と Batch 1→2 の間に ~5秒のアイドル区間を確認（GPU-Util 13%, 消費電力 56-62W）。クライアント側の `Await.result` → `Future` 投入のオーバーヘッド。3バッチで ~10秒のロス。

## 中間考察: 並列度と性能劣化

### 実測データまとめ（max_tokens=8192 統一）

| 並列度 | PP 平均 (tok/s) | TG 平均 (tok/s) | Phase 1 wall | content=0 |
|--------|----------------|----------------|-------------|-----------|
| 1 (C4) | 2,335 | 103 | 335s | 0/6 |
| 2 (C2) | 142 | 83 | 196s | 0/6 |
| 3 (C3) | 93 | 58 | 244s | 1/6 |

### 観察

1. **PP は並列度に対して超線形に劣化**: 1→2 で 16倍、2→3 でさらに 1.5倍。メモリ帯域競合が非線形に効いている
2. **TG も劣化するが PP より穏やか**: 1→2 で 1.2倍、2→3 で 1.4倍
3. **Wall time の最適値は並列度=2**: C2(248s) < C3(306s) < C4(385s)
4. **並列度を上げすぎると content=0 リスクが増加**: Thinking の completion tokens が並列環境で膨張する傾向

### 予備実験との PP 差異（OPEN）

予備実験（max_tokens=4096）では 4 並列で PP=388-1,722 tok/s だったのに対し、今回（max_tokens=8192）の 2 並列で PP=123-159 tok/s と大幅に悪い。C5（max_tokens=8192 での完全並列）で予備実験との直接比較が可能になる。

候補仮説:
- **max_tokens 依存**: 生成予約バッファが大きいほど KV キャッシュ帯域を圧迫
- **サーバ状態差**: プロンプトキャッシュ蓄積、Q4_K_M→Q4_K_XL 変更
- **計測方法差**: サーバ報告値 vs クライアント側計測

### llama-server ログ分析による再評価（重要）

サーバログ（190KB, 1993行）の詳細分析で、**プロンプトキャッシュが実験間で共有されていた**ことが判明。

#### プロンプトキャッシュの交絡

C4 → C2 → C3 → C5 をサーバ再起動なしで連続実行したため:
- C4 で作成されたキャッシュチェックポイント（各 ~63 MiB、8192 token 間隔）が残存
- C2/C3/C5 で同一プロンプト（同一法令の同一パート）を投入 → **キャッシュヒット**
- C5 のログで多数のタスクが `prompt_tokens=4`（キャッシュから復元、差分 4 tokens のみ処理）
- PP 速度は「4 tokens を処理した速度」として報告 → 小さい値になるのは当然

**含意**: C2-C5 の PP 速度値は「キャッシュヒット後の差分処理速度」であり、「プロンプト全体をゼロから処理した速度」ではない。C4 との PP 速度の直接比較は不正確。

#### SWA キャッシュ破損（C5）

C5 ではプロンプトキャッシュが 7,350 / 8,192 MiB（90%）に到達し、キャッシュ破損が発生:
- Tasks 77840-77842: 全チェックポイント無効化（SWA / hybrid recurrent memory との非互換）
- "forcing full prompt re-processing due to lack of cache data" がログに記録
- 破損後に全再処理が走ったタスクでは PP=2,079 tok/s に回復（単独実行相当）

#### `kv_unified=true` の帯域共有

4 スロットが共有 KV キャッシュに同時アクセスすることで CUDA メモリバス帯域が分割される。ただし、PP の「16倍劣化」の大部分はキャッシュヒットによる計測アーティファクトの可能性がある。

#### 実験の有効性について

PP 速度の絶対値にキャッシュの交絡がある一方、**wall time とTG 速度は影響を受けない**（TG はキャッシュと無関係な autoregressive decoding）。したがって:

- **Wall time の比較は有効**: C2(248s) < C5(283s) < C3(306s) < C4(385s) は信頼できる
- **TG 速度の比較は有効**: 並列度↑で TG 劣化（103→83→58→62 tok/s）は実測通り
- **PP 速度の絶対値は交絡あり**: キャッシュなしでの再実験が必要
- **content=0 の観察は有効**: Thinking 膨張は並列度の実効的な影響

#### 再実験時の推奨

1. サーバ再起動（またはプロンプトキャッシュ無効化: `--cache-ram 0`）後に各条件を独立実行
2. もしくは `--n-parallel 2` でサーバを起動し、C2 相当を基準として再計測

### 問題

2並列（C2）の PP が 123-159 tok/s。C4（逐次）の 2,244-2,409 tok/s から **16倍劣化**。予備実験の 4 並列（388-1,722 tok/s）よりもさらに悪い。

### 候補仮説

1. **max_tokens 依存**: 今回 8192 vs 予備実験 4096。生成予約バッファが KV キャッシュ帯域に影響？
2. **Thinking 生成パターン**: max_tokens 拡大で Thinking が長くなり、KV キャッシュの書き込みパターンが変化？
3. **scala-cli vs Python の HTTP クライアント差**: リクエストのシリアライゼーション・デシリアライゼーションのオーバーヘッド？
4. **サーバ状態**: 予備実験とサーバの再起動状態が異なる（プロンプトキャッシュの蓄積等）？
5. **計測方法**: サーバ報告の `prompt_per_second` vs 予備実験のクライアント側計測 → 今回はサーバ報告値なのでこの差ではない

### 検証方針

- C3 の結果で 3 並列の PP を確認 → 並列度との相関を見る
- max_tokens=4096 での C2 再実行 → max_tokens 依存性を切り分け
