# Stage EX: 法令要約戦略の比較実験 — Results

## 1. Introduction

法令全文の LLM 要約における最適な並列処理戦略を特定するため、民法（五編 + 附則、~122K prompt tokens）を対象に、Map-Reduce 方式の並列度を変化させた4条件を比較した。

### 実験条件

| 条件 | 戦略 | 並列度 | バッチ構成 |
|------|------|--------|-----------|
| C4 | Map-Reduce | 逐次（1） | 6パート×1 |
| C2 | Map-Reduce | 2並列×3バッチ | [0,1] [2,3] [4,5] |
| C3 | Map-Reduce | 3並列×2バッチ | [0,1,2] [3,4,5] |
| C5 | Map-Reduce | 完全並列（4スロット） | [0,1,2,3,4,5] |

全条件共通: `max_tokens=8192`, `temperature=0.0`, Qwen3.5-35B-A3B-UD-Q4_K_XL, `-c 262144`, `n_parallel=4`

C1（全文一発）は max_tokens=8192 での再実験が未実施のため、本レポートでは Map-Reduce 条件間の比較に焦点を当てる。

## 2. Methods

### 2.1 実装

scala-cli スクリプトとして実装。共通ユーティリティ（`common.scala`）を `//> using file` で参照する構成。

- 法令全文: e-Gov API V1 `/lawdata/129AC0000000089` で取得
- テキスト抽出: XML から `<Sentence>`, `<ArticleCaption>`, `<ArticleTitle>` を構造マークアップ付きで抽出
- Phase 1: 各編を個別に LLM で要約
- Phase 2: 6編の要約を統合して全体要約を生成
- 計測: llama-server 報告の PP/TG 速度、wall time、usage tokens を記録
- Thinking ブロック（`reasoning_content`）を全フェーズで保存

### 3.2 PP 速度の並列度依存性

| 並列度 | PP 平均 (tok/s) | C4 比劣化率 | TG 平均 (tok/s) | C4 比劣化率 |
|--------|----------------|------------|----------------|------------|
| 1 (C4) | 2,335 | 1.0x | 103 | 1.0x |
| 2 (C2) | 142 | **16.4x** | 83 | 1.2x |
| 3 (C3) | 93 | **25.1x** | 58 | 1.8x |
| 4+ (C5) | 100 | **23.4x** | 62 | 1.7x |

PP（Prompt Processing）は並列度に対して超線形に劣化。1→2 並列で 16 倍劣化という極端な落ち込みが発生。TG（Token Generation）の劣化は相対的に穏やか（1.2-1.8 倍）。

### 3.3 content=0 問題と completion tokens

## 4. Discussion

### 4.1 2並列×3バッチ（C2）がパレート最適

速度と品質の両面で C2 が最適:
- **速度**: C4 比 36% 短縮（385s → 248s）。C5 より 12% 速い
- **品質**: content=0 ゼロ、Phase 2 正常完了。C4 と同等の品質完全性
- **なぜ C5 より速いか**: C5 は Phase 1 が 194s と最短だが、2 パート欠損により Phase 2 の Thinking が膨張（90s vs C2 の 52s）。total では C2 が優位

### 4.2 PP 劣化の超線形性

1→2 並列で PP が 16 倍劣化するのは直感に反する。仮説:
- **VRAM 帯域の奪い合い**: Hybrid Attention-SSM アーキテクチャの attention 層（10/40層）が複数スロットで同時にアクセスされると帯域が飽和
- **max_tokens=8192 の影響**: 予備実験（max_tokens=4096）では 4 並列で PP=388 tok/s だったのに対し、今回（max_tokens=8192）の 2 並列で PP=142 tok/s。max_tokens 拡大が KV キャッシュ予約バッファを増大させ、帯域競合を悪化させている可能性
- **ただし wall time は確実に短縮**: PP 劣化にもかかわらず、2 パート並列の実効スループットは逐次の ~1.6 倍

### 4.3 Thinking 膨張と content=0

並列度が高いほど completion tokens が増加し、content=0 リスクが上昇する。推定メカニズム:
- 並列時の TG 速度低下（103→58-62 tok/s）により、同じ「考える量」を出力するのにより多くの wall time が必要
- あるいは、TG 低速環境でモデルの Thinking パターンが変化（より冗長な推論を生成）
- max_tokens=8192 でも 3 並列以上で不足。16384 への拡大で対処可能だが、TG 時間がさらに延びるトレードオフ

### 4.4 設計指針: `get_law_summary` ツールへの反映

| パラメータ | 推奨値 | 根拠 |
|-----------|-------|------|
| 並列度 | **2** | Wall time 最短（C2 = 248s）かつ品質完全 |
| max_tokens | **8192** | C2 で content=0 ゼロ。16384 は TG 時間増大のリスク |
| バッチ構成 | 3バッチ×2パート | 編の自然順序で分割 |
| Phase 2 | 単独実行 | PP/TG 共に単独実行速度に回復 |

### 4.5 実験の限界: プロンプトキャッシュの交絡

サーバログの事後分析で、C4 → C2 → C3 → C5 をサーバ再起動なしで連続実行したため、**プロンプトキャッシュが実験間で共有されていた**ことが判明した。

- C4（最初に実行）でキャッシュチェックポイントが作成される
- C2/C3/C5 では同一プロンプトがキャッシュヒット → PP 処理が大幅に省略
- C5 のログでは多数のタスクが `prompt_tokens=4`（キャッシュ復元、差分 4 tokens のみ処理）
- **PP 速度値の条件間比較は不正確**（C4 はキャッシュなし、C2-C5 はキャッシュあり）

ただし以下は影響を受けない:
- **Wall time**: PP がキャッシュで高速化されている分、C2-C5 の wall time は「キャッシュありの best case」
- **TG 速度**: autoregressive decoding はキャッシュと無関係。並列度による劣化は実測通り
- **content=0 問題**: Thinking 膨張は並列度の影響であり、キャッシュとは独立

加えて、C5 ではプロンプトキャッシュが上限（8,192 MiB）の 90% に達し、SWA/hybrid recurrent memory との非互換によるキャッシュ破損が発生。一部タスクで全再処理が走った。

**結論**: C2 がパレート最適という結論は wall time + 品質ベースで有効だが、PP 速度の劣化メカニズムの解明にはキャッシュを無効化した再実験が必要。

### 4.6 Deferred: 検証価値はあるが現時点では ROI が低い課題

以下は無意味ではないが、現時点の実験精緻化として行うより、ツール実装後の実運用フィードバックに基づいて検証する方が効率的と判断した課題。

#### インフラ最適化系

| 課題 | 意義 | Defer 理由 |
|------|------|-----------|
| **キャッシュ無効化での再実験** | PP 速度の交絡を除去し、並列度と PP の正確な関係を確定 | C2 最適の結論は wall time + TG ベースで有効。PP の絶対値が設計判断を変える見込みは低い |
| **KV キャッシュ量子化** (`--cache-type-k q8_0`) | 帯域圧力半減による並列 TG 改善の可能性 | サーバ全体に影響（要約以外のツールも）。法令テキストの精度要求下での品質劣化検証コストが高い。実運用で性能問題が顕在化してから検証する方が的確 |
| **`--n-parallel 2`** | スロット数をアプリケーション最適値に合わせる | クライアント側で既に並列度制御済み。REPL 通常使用（1リクエスト）に影響しないため実益が薄い |
| **C1（全文一発）max_tokens=8192 再実験** | 予備実験との厳密な比較 | 予備実験（81s）で実用性は確認済み。ツール実装で C1 パスも用意するため、その時点で計測可能 |

#### 品質改善系

| 課題 | 意義 | Defer 理由 |
|------|------|-----------|
| **System prompt へのメタデータ付与** | 法令名・全体構成・担当パート範囲を提供し、Thinking の探索空間を狭める → content=0 軽減・条番号正確性向上 | **ROI は高い**が、`get_law_summary` ツール実装に組み込んで検証する方が実践的。机上実験より実運用フィードバックが速い |
| **分割戦略の最適化** | 意味論的 2 分割（財産法/家族法）、章単位、LLM 自律分割 | 探索空間が広い。現在の編単位分割で実用品質が確認されており、最適化は段階的に行うべき |
| **System vs User 配置の A/B テスト** | Qwen3.5 の System 遵守特性を活かした最適配置の確定 | メタデータ付与と同時にツール実装で検証可能 |

### 4.7 実験クローズ判定

本実験（Round 1）で得られた設計根拠は `get_law_summary` ツール実装に十分:
- **戦略**: Map-Reduce（C2: 2並列×3バッチ）をデフォルト、全文一発をオプション
- **パラメータ**: max_tokens=8192, temperature=0.0
- **品質制御**: System prompt に条番号正確性指示 + メタデータ付与（実装時に組み込み）
- **既知のリスク**: 並列度3以上で content=0 が発生。2並列で回避

Deferred 課題はツール実装・実運用フェーズで段階的に検証する。
