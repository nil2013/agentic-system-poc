## 1. Introduction

### 1.1 目的

LLM が「自分で答えるか、外部ツールを呼ぶか」を判断するエージェントループ（ReAct パターン）を実装し、tool calling の動作を検証する。2つのパス（OpenAI 互換 / プロンプトベース）を比較する。

### 1.2 予定していた作業

| 作業 | ガイド参照 |
|------|-----------|
| ツール関数（searchStatute）の実装・単体テスト | §2.2 |
| パス A: OpenAI 互換 tool calling エージェントループ | §2.3 |
| パス B: プロンプトベース tool calling（フォールバック） | §2.4 |
| 5問のテストケースで両パスを比較 | §2.5 |

### 1.3 仮説（ガイドの予想）

- パス A が推奨だが、llama-server の tool calling 対応状況に依存
- パス B はフォールバック
- 5問中、質問 5（複数ツール呼び出し）が最も難しい

### 1.4 Stage 1 からの持ち越し

- Thinking mode 対策: `max_tokens=4096` に引き上げ（ガイドは 1024 だが、環境適応として。→ NOTES.md 参照）
- SystemPrompt でのスキーマ遵守率が高いことが確認済み（Stage 1）

## 2. Methods

### 2.1 ツール定義

`searchStatute(lawName, articleNumber)`: e-Gov 法令 API V1 から指定法令の指定条文を取得。対応法令は民法・刑法・憲法・行政手続法・行政事件訴訟法・民事訴訟法の6法。

### 2.2 テストケース（ガイド §2.5）

| # | 質問 | 期待される動作 |
|---|------|--------------|
| 1 | 「民法709条の条文を教えてください」 | ツール呼び出し → 条文取得 → 提示 |
| 2 | 「今日の天気は？」 | ツールを呼ばずに直接回答 |
| 3 | 「不法行為の損害賠償請求の根拠条文は？」 | ツール呼び出し（709条）→ 取得 → 説明 |
| 4 | 「刑法の殺人罪は何条ですか？条文も示してください」 | ツール呼び出し（199条）→ 取得 → 提示 |
| 5 | 「民法1条と709条の両方を見せてください」 | 2回のツール呼び出し |

### 2.3 評価指標

- **ツール呼び出し正否**: 呼ぶべき質問で呼べたか（再現率）、不要な質問で呼んでいないか（適合率）
- **引数の正確性**: law_name, article_number が正しいか
- **最終回答の品質**: ツール結果を踏まえた回答が返っているか

### 2.4 パラメータ

- `max_tokens`: 4096（thinking mode 対策）
- `temperature`: 0.0
- `MAX_TOOL_ROUNDS`: 5

### 2.5 実装

- `statute_search.scala`: ツール関数 + 単体テスト
- `agent_path_a.scala`: パス A（OpenAI 互換 tool calling）
- `agent_path_b.scala`: パス B（プロンプトベース）

### 3.1 ツール単体テスト

全ケース正常動作。

| テスト | 結果 |
|--------|------|
| 民法709条 | 条文テキスト取得 OK |
| 刑法199条 | 条文テキスト取得 OK |
| 不存在法令 | エラーメッセージ返却 OK |

### 3.2 パス A: OpenAI 互換 tool calling

| # | ツール呼び出し | 引数 | 回答品質 | 判定 |
|---|--------------|------|---------|------|
| 1 | OK（呼んだ） | 民法, 709 ✅ | 条文提示 + 説明 | **OK** |
| 2 | OK（呼ばなかった） | — | 「機能外」と適切に回答 | **OK** |
| 3 | OK（間接的に推論して呼んだ） | 民法, 709 ✅ | 根拠条文として提示 | **OK** |
| 4 | OK（呼んだ） | 刑法, 199 ✅ | 条文提示 + 解説 | **OK** |
| 5 | OK（2回呼んだ） | 民法, 1 + 民法, 709 ✅ | 両条文を提示 | **OK** |

**5/5 全問正解。** テスト 5 では **1ラウンドで parallel tool_calls**（2つの tool_call を同時発行）を実現。

### 3.3 パス B: プロンプトベース tool calling

| # | ツール呼び出し | 引数 | 回答品質 | 判定 |
|---|--------------|------|---------|------|
| 1 | OK | 民法, 709 ✅ | 条文提示 + 要件分析 | **OK** |
| 2 | OK（呼ばなかった） | — | 「機能外」と適切に回答 | **OK** |
| 3 | OK | 民法, 709 ✅ | 条文提示 + ポイント解説 | **OK** |
| 4 | OK | 刑法, 199 ✅ | 条文提示 + 解説 | **OK** |

**4/5 正解。** テスト 5 でツール呼び出しが発生せず、content も空。

### 3.4 パス比較

| 観点 | パス A | パス B |
|------|-------|-------|
| single tool calling | 5/5 ✅ | 4/4 ✅ |
| multi tool calling | 1/1 ✅（parallel） | 0/1 ❌ |
| ツール不要の判断 | 1/1 ✅ | 1/1 ✅ |
| 間接的推論 | 1/1 ✅ | 1/1 ✅ |
| **総合** | **5/5** | **4/5** |

### 3.5 遭遇した問題

1. **Thinking mode と max_tokens**: ガイドの `max_tokens=1024` では不足。4096 に引き上げて解決（Stage 1 と同じ問題）
2. **パス B のテスト 5 失敗**: プロンプトベース方式は1回の応答で1つの tool_call JSON しか返せない設計。2つの条文を同時に要求するテスト 5 ではモデルが対処できなかった。パス A の parallel tool_calls との構造的差異
3. **scala-cli 複数ファイル warning**: `//> using` ディレクティブが複数ファイルにあると warning。Stage 3 の sbt 移行で解消予定

## 4. Discussion

### 4.1 ガイドの予想との比較

ガイドは「パス A が推奨、パス B はフォールバック」としていたが、パス A が **完璧に動作した**ため、フォールバックの出番がなかった。

ガイドの懸念事項（§2.1）:
- ❌ **「llama-server の tool calling が不安定」**: 実際は非常に安定。parallel calling まで動作
- ❌ **「Uncensored バリアントの tool calling 品質」**: Q4_K_M 量子化でも問題なし（ただし公式 instruct 版なので Uncensored の懸念は該当しない）

### 4.2 パス B の限界と Stage 1 との接続

Stage 1 で「SystemPrompt のスキーマ遵守率が高い」ことを確認し、パス B が実用的と予想した。実際、**single tool calling ではパス B も完璧に動作**した。しかし、**multi tool calling（テスト 5）で構造的限界が露呈**:

- パス A: API レベルで `tool_calls` 配列を返すため、1ラウンドで複数ツールを parallel に呼べる
- パス B: content に JSON を1つ返す設計。複数呼び出しにはループの逐次処理が必要だが、「まず1つ目を呼ぶ」という判断自体ができなかった

### 4.3 Stage 3 への示唆

- パス A の採用で問題なし。パス B は「パス A が使えない環境」のフォールバックとしてのみ意味がある
- parallel tool_calls が動作するため、Stage 3（複数ツール）でも同一ラウンドで異なるツールを呼べる可能性がある
- `max_tokens=4096` は引き続き必要（thinking mode）

### 4.4 追加検証の候補（未実施）

- パス B で逐次的な複数呼び出し（プロンプトを工夫して「まず1つ目を呼べ」と指示）
- Q6_K vs Q4_K_M での tool calling 精度比較
- temperature > 0 での tool calling 安定性
