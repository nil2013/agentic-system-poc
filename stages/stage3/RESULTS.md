## 1. Introduction

### 1.1 目的

3つのツール（条文検索・法令名検索・計算）を LLM に提示し、質問に応じた適切なツール選択（ルーティング）ができるか検証する。description の粒度がルーティング精度に与える影響を比較する。

### 1.2 予定していた作業

| 作業 | ガイド参照 | 実施状況 |
|------|-----------|---------|
| findLawByKeyword, calculate ツール追加 | §3.1 | ✅ 実施 |
| description 2パターンでルーティング精度比較 | §3.2 | ✅ 実施 |
| メッセージ型の ADT 化 | §3.3 | ⏭ 延期（sbt 移行時に実施） |

### 1.3 仮説（ガイドの予想）

- 簡潔な description より詳細な description の方がルーティング精度が高い
- 「descriptionの書き方がルーティング精度を支配する」

## 2. Methods

### 2.1 ツール一覧

| ツール | 機能 |
|--------|------|
| searchStatute(lawName, articleNumber) | 法令の条文を取得（Stage 2 から流用） |
| findLawByKeyword(keyword) | キーワードで法令名を検索（e-Gov API lawlists） |
| calculate(expression) | 四則演算（自前の再帰下降パーサ。→ NOTES.md 14:20） |

### 2.2 テストケース（ガイド §3.2）

| # | 質問 | 正しいツール |
|---|------|------------|
| 1 | 「個人情報に関する法律にはどんなものがありますか？」 | findLawByKeyword |
| 2 | 「民法709条の条文を見せてください」 | searchStatute |
| 3 | 「研究費140万円の30%はいくらですか？」 | calculate |
| 4 | 「消費者契約法の正式名称は？」 | findLawByKeyword |
| 5 | 「刑法199条を教えてください」 | searchStatute |
| 6 | 「50000 + 150000 + 400000 の合計は？」 | calculate |

### 2.3 Description パターン

| パターン | searchStatute | findLawByKeyword | calculate |
|---------|--------------|-----------------|-----------|
| 1（簡潔） | 「日本の法令の条文を検索する」 | 「法令名を検索する」 | 「計算する」 |
| 2（詳細） | 引数説明・例示付き | 用途説明・例示付き | 用途説明・例示付き |

### 2.4 ガイドからの逸脱

- **calculate の実装**: ガイドは `javax.script.ScriptEngineManager` を使用するが、JVM 20 に JS エンジンがないため自前の再帰下降パーサで代替（→ NOTES.md 14:20）
- **ADT 化の延期**: メッセージ型の ADT 化（§3.3）は sbt プロジェクト移行時に実施する方が整理しやすいため延期

## 4. Discussion

### 4.1 ガイドの予想との比較

ガイドの中核仮説「descriptionの書き方がルーティング精度を支配する」は、本実験では**検証できなかった**。両パターンで 100% の正答率であり、差が観察されない。

### 4.2 天井効果の再現

Stage 1（構造化出力 100%）→ Stage 2（tool calling 5/5）→ Stage 3（ルーティング 6/6）と、一貫して天井効果が出ている。原因の候補:

1. **モデル能力**: Qwen3.5-35B-A3B（BFCL-V4: 67.3）の tool calling 能力が、これらのタスク難易度に対して十分すぎる
2. **ツール間のカテゴリ分離が明確**: 条文検索 / 法令名検索 / 計算 は意味的に十分離れている。description がなくても関数名だけでルーティング可能と推測
3. **temperature=0**: 確定的出力で robustness のバリエーションが観測されない

### 4.3 description 精度差が顕在化する条件（推測）

- ツール間の境界が曖昧なケース（例: 「民法の構成は？」→ searchStatute? findLawByKeyword?）
- ツール数が多い場合（3つ → 10+）
- モデルサイズが小さい場合（9B、4B）

### 4.4 ADT 化の延期について

ガイド §3.3 のメッセージ型 ADT 化は、scala-cli の単一ファイル実行ではファイル分割の恩恵が薄い。sbt プロジェクト移行（ガイドの Stage 3+ 方針）と同時に実施する方が自然。設計自体はガイドの `enum ChatMessage` をそのまま使える。

### 4.5 Stage 4 以降への示唆

- ルーティング精度はこの環境では問題にならない。Stage 4（状態管理）・Stage 5（計画）ではより高レベルの判断品質が問われる
- 複数ステージを通じた天井効果は、**Q4_K_M vs Q6_K の比較検証で差が出にくい可能性**を示唆する
