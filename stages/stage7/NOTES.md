### 重要な発見

**1. ツール呼び出しターンでは thinking が返らない**

T1（ツール呼び出しあり）と T3（ツール呼び出しあり）で thinking chars = 0。これは AgentLoop のループ構造に起因: tool_calls が含まれるレスポンスでは content が null/空で reasoning_content も空。thinking は最終回答（tool_calls なし）のレスポンスでのみ返される。

→ つまり **ツール選択の推論過程はキャプチャできない**。ツール呼び出しの判断は「1回目の API 呼び出し」で行われ、そのレスポンスは tool_calls を含むため content/reasoning_content が空になる。最終回答時の thinking はツール結果を踏まえた「回答生成の推論」であり、ツール選択の推論ではない。

**2. T2（天気）の thinking は正しい判断の推論過程**

「ツールを使う必要がない」という判断の推論過程が見える:
- 「日本法に関する質問ではない」
- 「提供ツールは法律関連のみ」
- → ツール不使用を選択

**3. T4（非実在法令）で content が空**

thinking chars = 335 あるが content = 0。thinking が max_tokens を消費しきった。ただし thinking の内容は「皇族に対する尊崇義務を定めたものではない」「皇室典範や皇室に関する法律を確認すべき」という正しい分析。content に到達する前にトークンが尽きた。

**4. T5（複雑タスク）で thinking は回答構造の計画**

thinking の内容: 「3つの法律を取得した。次にこれらの目的規定を比較する」。これは回答生成の計画であり、ツール選択の推論ではない。

### thinking キャプチャの構造的限界

現在の AgentLoop の構造（tool_calls ありのレスポンスでは reasoning_content が空）により、**ツール選択の推論過程は観察不能**。観察できるのは:
- ツール不使用の判断過程（T2）
- 最終回答生成時の推論（T5）
- ツール結果を踏まえた分析（T4）

ツール選択の推論を観察するには、AgentLoop の「1回目の API 呼び出し」のレスポンスから reasoning_content を抽出する必要がある。現在の実装では中間ラウンドの reasoning はキャプチャしていない。

## 2026-03-24 18:35 — 重大な修正: reasoning_content は API から返されていた

### 発見

curl で直接確認したところ、**llama-server は tool_calls レスポンスでも `reasoning_content` を返している**:
```json
{
  "has_tool_calls": true,
  "content": "",
  "reasoning_content": "ユーザーは民法709条の条文を求めています。get_article関数を使用して..."
}
```

**ツール選択の推論過程が含まれている**（「get_article を使用」という判断の根拠）。

### 原因

Stage 7 RESULTS.md §3.3 で「ツール呼び出しターンでは reasoning_content が空（llama-server の挙動）」と報告したが、**これは誤り**。

正確な原因は **AgentLoop の設計**:
- 中間ラウンド（tool_calls あり）: `assistantMsg` は `currentMessages` に追加されるが、`reasoning` は `TurnResult` に含まれない
- 最終回答（tool_calls なし）: `assistantMsg.reasoning` を `TurnResult.reasoning` に設定

→ `TurnResult` は最終回答の reasoning のみを返し、中間ラウンドの reasoning は**メッセージ履歴内には存在するが、呼び出し元に公開されていなかった**。

### 影響

- Stage 7 の「ツール選択の推論は観察不能」は**実装の制約であり、API の制約ではない**
- AgentLoop を修正して中間ラウンドの reasoning を収集すれば、ツール選択の推論過程が完全に観察可能
- 発展的課題 A7-3（中間ラウンド reasoning キャプチャ）は API 仕様の制約ではなく実装の改修で対応可能

### T4 の content=0 問題の追加分析

Round 2 のログを詳細確認したところ、T4 の thinking ブロック内に以下が含まれていた:

```
</function>

この法律は「皇族の身分を離れた者及び皇族となつた者の戸籍に関する法律」であり、
皇族に対する尊崇義務を定めたものではありません。
（以下、完全な回答テキスト）
```

**回答テキストは生成されていたが、`reasoning_content` フィールドに混入していた。** `</function>` タグの残骸が thinking 内に含まれており、llama-server（または Qwen3.5 のテンプレート）が thinking と content の境界を正しくパースできていない。

→ R1-2 の T4 content=0 は「LLM が回答しない判断をした」のではなく、**llama-server の thinking/content 境界パースの問題**だった可能性が高い。R3 で同じクエリが content=177 で正常に返っているのは、実行ごとのパース挙動の揺らぎ、または max_tokens/context 設定の差異による影響。
