## 2026-03-25 11:17 — REPL 起動時の Warning と日本語入力問題

### 症状

1. `scala.runtime.LazyVals$` / `sun.misc.Unsafe` 警告 → Scala 3.6.4 + JDK 25 の組み合わせ問題。キーボード問題とは無関係
2. `java.lang.System::load` / native access 警告 → JDK 24+ の制限。`--enable-native-access=ALL-UNNAMED` で抑制可能
3. **`Unable to create a system terminal, creating a dumb terminal`** → **主因**

### 原因分析

sbt の `connectInput := true` は stdin のバイト転送のみで、**TTY（制御端末）の継承は保証しない**。JLine は OS の制御端末（system terminal）を必要とするが、sbt fork 経由ではそれが取得できない。

dumb terminal にフォールバックすると JLine の行編集機能（カーソル移動、マルチバイト対応 BackSpace、入力履歴）が全て無効になる。

### 対応方針（優先順位順）

1. **sbt 以外での直接起動で切り分け**: これで直るなら sbt fork + TTY 継承が原因確定
2. **JLine 依存を明示化 + バージョン更新**: umbrella artifact → terminal/reader/terminal-jni 分離、3.28→3.30 系
3. **TerminalBuilder 設定改善**: `dumb(false)` で失敗を明示化、`provider("jni")`, `encoding(UTF_8)` 指定
4. **`--enable-native-access=ALL-UNNAMED`** を javaOptions に追加
5. **Scala/JDK 組み合わせ是正**: JDK 25 のサポート下限は Scala 3.7.1。3.6.4 は非推奨帯

### 補足: Scala 3.6.4 + JDK 25 の互換性

公式互換表では JDK 25 のサポート下限は Scala 3.7.1。3.6.4 は「動くことはあるが推奨されない」。`LazyVals` の `Unsafe` 依存は Scala 3.8 で解消。ただしこれはキーボード問題の直接原因ではない。

### 民訴196条のハルシネーション事例

**経緯:**
1. 「訴訟に関する記録を見るのに使えそうな条文」と質問
2. LLM が `get_article("民事訴訟法", "195")` を呼び出し → 受命裁判官の証人尋問（無関係）
3. content 空で返却
4. 「前の回答を再現して」→ Tool なしで「196条が訴訟記録の閲覧」と回答（ハルシネーション）
5. 「きちんとチェックして」→ 196, 197, 198, 247, 248 を5連続で取得試行 → MAX_TOOL_ROUNDS 超過
6. ユーザーが「91条を見て」と教示 → 正しい条文（91条）を取得して回答

**正解**: 民事訴訟法 **第91条**（訴訟記録の閲覧等）

### 原因の分析（ディスカッション）

3つの要因が複合:

1. **ツール機能の欠如（主因、50%）**: 「法令内の条文をキーワードで検索する」ツールがない。`find_laws` は法令名の検索のみ、`get_article` は条番号指定のみ。LLM は「閲覧に関する条文は何条か」を推測するしかない構造。**推測しか手段がない状況ではハルシネーションは構造的に誘発される**

2. **LLM のハルシネーション（30%）**: 196条を「訴訟記録の閲覧」と誤同定。ただしツールがない以上推測は不可避であり、ハルシネーションの「責任」はツール不足と共有される

3. **MAX_TOOL_ROUNDS の打ち切り（20%）**: 5ラウンドで探索が中断。仮に回数を増やしても、正しい条番号（91条）に到達できたかは疑わしい（500条超の法令を総当たりはできない）

### FallbackControl の逆効果の可能性

「前の回答を再現して」で Tool なしに回答した流れ:
- FallbackControl（「内部知識で補完するな」）が効いている状態で、ツールで答えが見つからなかった
- 最初の応答は content 空（FallbackControl が機能して内部知識での補完を抑制？）
- しかし「再現して」と言われたら FallbackControl の制約が緩んで内部知識ベースで回答
- **「再現して」というユーザー指示が FallbackControl を override した**可能性

### Stage EX への示唆

**最優先で必要なツール**: 法令内条文のキーワード検索（`search_articles("民事訴訟法", "閲覧")` → 91条がヒット）。これがあれば推測→ハルシネーションの構造的問題が解消される。

e-Gov API V1 には条文内容検索のエンドポイントがないため、`/lawdata/{lawId}` で法令全文を取得し、XML 内を検索する方式が現実的。ただしレスポンスが大きい（民法で 1.7MB）のでキャッシュ戦略が重要。

## 2026-03-25 12:21 — 民事訴訟法の構造探索テスト（new_session）

### 概要

新規セッション `new_session` で民事訴訟法の全体構造を、限られたツール（`get_article` の条番号指定のみ）を使って探索させた。11ターンの対話で、LLM が自発的に二分探索的アプローチを採用し、法令の条文構造を段階的に推定した。

### 発見（ユーザー視点）

1. **Tool Call 回数のプロンプト制御は効かない**: 「3〜5個引いて」と指示しても MAX_TOOL_ROUNDS=5 に引っかかる
2. **計画→実行パターンは遵守される**: 計画を立てさせてから「その通りにやれ」と指示すると条文数制限を遵守
3. **限られたツールでも構造推定ができる**: 条番号指定しかできないツールで、民訴全体（~405条）のマップを構成

### 発見（Claude 視点）

4. **MAX_TOOL_ROUNDS=5 が真のボトルネック**: `find_laws(1) + get_article(4) = 5` で上限ぴったり。ユーザーの「3個まで」制限は `find_laws(1) + get_article(3) = 4` で収まる — 無意識のワークアラウンド

5. **二分探索の自発的採用**: LLM が 400→450(なし)→410(なし) と区間を狭める探索戦略を誰にも教わらず自発的に採用。Adaptive Planning の良質な実例

6. **Plan-then-Execute の「再発明」**: 「計画立てて→次のターンで実行」はまさに PtE。Stage 5 で失敗した依存引数問題が起きないのは、**計画時点で条文番号が確定している**から。「PtE は引数が事前確定できるタスクでは有効」— Stage 5 の結論を修正すべき知見

7. **「中間報告」パターン = implicit self-evaluation**: ユーザーが「中間報告して」と求めるたびに、LLM が「分かっていること/分かっていないこと」を整理する。Stage 6 の self-evaluation をユーザーが手動発動させている構造。evaluator の thinking 消費問題もない（回答として content に入るため）

8. **条文構造推定の精度**: 実際の民事訴訟法は全7編 405条。LLM の推定「約402〜409条」は相当正確。編の区切りは粗いが、大構造（総則→第一審→上訴→再審→特別手続→督促→執行停止）は概ね捕捉

### Stage EX への示唆

- **MAX_TOOL_ROUNDS の動的調整**: 探索的タスクでは 5 では足りない。タスクの性質に応じて 10-15 に引き上げるか、ユーザーが設定できるコマンド（`/max-tools N`）が必要
- **法令構造表示ツール（EX-1）**: 章・節の目次一覧を返すツールがあれば、二分探索を省略して一発で全体構造が得られる
- **条文内容検索ツール（EX-1）**: 「閲覧に関する条文は？」に答えるために必須
- **Plan-then-Execute の適用条件の整理**: PtE が有効なケース（引数が事前確定可能）と無効なケース（前ステップの結果に依存）を SystemPrompt で案内すると、LLM が適切な戦略を選べるかもしれない
- **「中間報告」の自動発動**: N ターンごとに自動的に「ここまでの理解をまとめて」を挿入する仕組み。Stage 6 の evaluator とは異なるアプローチで回答品質を維持
